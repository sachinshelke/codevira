# Codevira v3.7.1 — release notes

**Released:** 2026-07-20
**Install:** `pipx install --upgrade codevira` or `pip install codevira==3.7.1`

---

## TL;DR

v3.7.1 is a reliability release. It fixes a migration bug that could strand a
project's entire decision memory, several defects that could bind a session to
the **wrong project**, and a cluster of read-layer bugs that made decision
memory *wrong* rather than lost. It also makes writes to your IDE config
non-destructive. The one new feature is `codevira init --shared` — opt-in
team-shared memory. All fixes are model-free and local. Full per-line history is
in [CHANGELOG.md](../../CHANGELOG.md#371--2026-07-20).

---

## Upgrade: automatic, no manual steps

Upgrading is automatic. Codevira migrates your memory on the **first server
start** after the upgrade — nothing to run by hand, and your existing decisions
stay put. The migration is now non-destructive: it never moves your memory and
never renames `.codevira/`, and a self-heal pass restores any memory that a
previous version may have stranded.

```bash
pipx upgrade codevira      # or: pip install --upgrade codevira
codevira --version         # should report 3.7.1
```

If a project was hit by the old migration bug before upgrading and its store was
git-tracked, recover it with `git checkout -- .codevira/`; otherwise the new
migration restores it automatically on the next start.

## What's fixed

- **Data loss / migration.** The old centralization migration copied derived
  files but not the memory itself, then renamed `.codevira/` away — so a
  migrated project could read as **zero decisions** while its real history sat
  stranded. Migration is now non-destructive, self-heals stranded memory on
  every start, and refuses to delete a backup that holds the only copy of a
  memory file. In-repo and team-shared stores are never migrated.
- **Wrong-project binding.** Several independent defects could make a session
  read or write **another project's** store — a pin that only lasted one tool
  call, a shared git remote hijacking an initialized project, a metadata-only
  "ghost" directory out-ranking a real store, unrecognized git worktrees and
  submodules, and a stale cache that never re-resolved. All are fixed; an
  explicit local init now outranks a remote match. A **bare** global `codevira`
  entry in `~/.claude.json` (no `--project-dir`) that used to hijack the binding
  is now displaced when a project-scoped entry is written, and `doctor` gained a
  `claude_binding_conflict` check.
- **Decision-memory correctness.** A retired decision now disappears everywhere
  (search, prompt injection, and the generated `AGENTS.md`), not just from
  search. Amendments no longer inject as blank noise or silently reset the
  180-day `do_not_revert` staleness clock. Decisions on paths with glob
  characters (e.g. `app/[slug]/page.tsx`) are findable and protectable.
  Bulk-imported decisions are now scrubbed for secrets, and superseding an
  already-superseded decision is refused instead of forking the chain.
- **Concurrency.** `repair_ids` — which runs at every server start — could
  overwrite a decision recorded in another IDE window during its
  read-then-write gap. It now reads, transforms, and rewrites under a single
  exclusive lock, and leaves a clean store byte-identical.
- **IDE config safety.** `~/.claude.json` holds your account, project history,
  and every other MCP server. A momentarily unparseable file used to be treated
  as "missing" and replaced wholesale; codevira now refuses to overwrite a
  config it cannot read.
- **Antigravity.** The server now **degrades gracefully** ("limited mode")
  instead of crashing when launched on an invalid root (the `/ is a system
  directory` / `initialize: EOF` crash), and Antigravity gets per-project
  entries instead of the crash-causing bare global one.

## New: `codevira init --shared` (team-shared memory)

Two engineers on the same repo need to see each other's decisions, which means
`.codevira/` memory has to stay committed.

- Run `codevira init --shared` to write `git_shared: true`. That keeps the
  project's decision memory **git-tracked**, and the built-in git merge driver
  reconciles concurrent edits so two people recording decisions at once don't
  clobber each other. The flag persists, so a plain re-init won't undo it.
- **The default (no flag) stays per-machine** — memory is not committed — so
  unrelated projects never bleed into each other. It's an opt-in flag, nothing
  more.

## Enforcement note (Antigravity)

Claude Code's `PreToolUse` hook remains the verified **hard block** — it refuses
a decision-reverting `Edit`/`Write` before it touches disk. In v3.7.1,
Antigravity also gets a `PreToolUse` enforcement hook that surfaces and can
block decision-reverting edits. (Its hard-block behavior is pending live-payload
verification; treat it as "can block" rather than a guaranteed veto for now.)

## If an IDE misbehaves after upgrade

Your memory lives in each project's in-repo `.codevira/` and is not lost by any
of this — but the IDE *binding* can point at the wrong place until you clean a
stale config entry. Two fixes cover most cases:

1. Remove a stray or temporary codevira entry with `codevira untrack <path>` (or
   sweep dead ones with `codevira clean --ghosts`).
2. Run `codevira doctor` — it names the project the server bound to and why, and
   ships the exact fix for each ⚠ / ✗.
